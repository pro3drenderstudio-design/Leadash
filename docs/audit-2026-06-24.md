# Leadash Codebase Audit — 2026-06-24

Comprehensive audit covering security, performance, reliability, and architecture across `apps/web`, `apps/worker`, migrations, and schema.

---

## CRITICAL (5)

### 1. Race condition in send batch — duplicate sends possible
**File:** `apps/web/src/lib/outreach/send-runner.ts:378`
Two concurrent cron invocations can claim the same enrollment before either updates it. The `OR next_send_at.is.null` condition prevents the index from being used atomically.
**Fix:** Use `SELECT ... FOR UPDATE SKIP LOCKED` in a stored procedure.

### 2. Campaigns can't auto-resume after upgrade
**File:** `apps/web/src/lib/outreach/send-runner.ts:191`
Campaigns paused with reasons like "Trial expired" or "Monthly limit reached" have no mechanism to unpause when conditions clear. Customers who upgrade have stuck campaigns until they manually resume.
**Fix:** When plan/billing status changes, enqueue a job to unpause campaigns with matching `pause_reason`.

### 3. Silent failures in all scheduler crons
**File:** `apps/worker/src/schedulers/index.ts:106`
Dozens of `catch { /* non-fatal */ }` blocks swallow errors with no logging. A database outage silently pauses all sends across all workspaces with no alerting.
**Fix:** Log all catch blocks at error level. Send error metrics to monitoring.

### 4. N+1 queries in send runner — ~600 queries per batch
**File:** `apps/web/src/lib/outreach/send-runner.ts:318`
For each enrollment: separate queries for the lead, sequence step, blacklist check, pause-after-open check, follow-up subject, and next step. At 100 enrollments = 600+ queries → Vercel 60s timeout → claimed but unsent enrollments.
**Fix:** Batch-load all leads at start with `.in("id", leadIds)` then use a map. Reduces ~600 queries to ~10.

### 5. Stale enrollment claims never released
No `claimed_at` column on enrollments. If a worker crashes after claiming but before sending, that enrollment is stuck forever.
**Fix:** Add `claimed_at timestamptz` column. Add a cleanup cron to release claims older than 2 hours.

## HIGH (6)

### 6. Orphaned FK rows from partitioned `outreach_sends`
Migration 027 explicitly drops FKs from `outreach_tracked_links` and `outreach_replies` to `outreach_sends`. No cleanup job. Orphaned rows accumulate at scale.
**Fix:** Add monthly cleanup job: `DELETE FROM outreach_tracked_links WHERE send_id NOT IN (SELECT id FROM outreach_sends)`.

### 7. Webhook handler accepts unvalidated JSON
**File:** `apps/web/src/app/api/support/inbound/route.ts:55`
Raw `req.json()` with no schema validation or size limit. Fields cast without sanitization, passed to DB inserts and email sends.
**Fix:** Use zod schema. Reject payloads > 1MB. Sanitize email addresses.

### 8. Admin impersonation doesn't check role
**File:** `apps/web/src/app/api/admin/users/[userId]/impersonate/route.ts`
Fetches `admin.role` but never uses it. Any row in `admins` table can impersonate any user regardless of role.
**Fix:** `if (!admin || admin.role !== "super_admin") return 403`.

### 9. Credit deduction is not atomic
**File:** `apps/worker/src/workers/lead-campaign-worker.ts:155`
Three separate DB calls: update balance → update campaign → insert transaction. Concurrent campaigns both read the same balance → lost updates. Crash after step 1 deducts credits with no record.
**Fix:** Wrap in a Supabase RPC/stored procedure that handles all three in one transaction.

### 10. Impersonation banner cookie is not HttpOnly
**File:** `apps/web/src/app/api/admin/users/[userId]/impersonate/route.ts:57`
`httpOnly: false` allows client-side JS to read admin identity. XSS risk.
**Fix:** Return banner state via API endpoint instead of accessible cookie.

### 11. No dead-letter queue for failed worker jobs
Jobs that exhaust retries silently disappear. Failed `processLeadCampaign` jobs leave no trace.
**Fix:** Configure DLQ per worker. Add monitoring cron to alert on growing DLQ size.

## MEDIUM (10)

### 12. Partial index missing for `next_send_at IS NULL` enrollments
The OR condition in the scheduler query can't efficiently use the B-tree index.
**Fix:** `CREATE INDEX idx_enrollments_due_null ON outreach_enrollments(workspace_id, status) WHERE next_send_at IS NULL`. Use two separate queries instead of OR.

### 13. `.env` file in working directory
Run `git log -p --all -- .env | head -100` to confirm it was never committed. If it was, rotate all secrets and use `git filter-repo`.

### 14. Duplicate lead campaign processor
Two implementations: `apps/worker/src/workers/lead-campaign-worker.ts` and `apps/web/src/lib/lead-campaigns/processor.ts`. Different retry logic, credit deduction, error handling.
**Fix:** Consolidate into a shared package.

### 15. No soft deletes
Campaigns, enrollments, leads are hard-deleted. No recovery, no audit trail.
**Fix:** Add `deleted_at timestamptz` to all main tables. Filter `WHERE deleted_at IS NULL`.

### 16. Hardcoded limits scattered across 8+ files
`MAX_DAILY_SENDS`, `VERIFY_CONCURRENCY`, `CREDITS_PER_LEAD`, `WORKSPACE_CHUNK_SIZE` etc.
**Fix:** Centralize in `apps/web/src/lib/config.ts`. Make some admin-configurable via `admin_settings`.

### 17. No input validation on API routes
`daily_cap` could be -1, `wait_days` could be 10000, `smtp_password` has no max length.
**Fix:** Use zod schemas on all request bodies.

### 18. Lead copy loop does 500 separate DB calls
**File:** `apps/web/src/lib/lead-campaigns/processor.ts:61`
50K leads = 500 sequential inserts. Fix: batch via RPC or `COPY`.

### 19. Monthly send counter reset locks table
**File:** `apps/worker/src/schedulers/index.ts:147`
Single UPDATE on potentially 100K workspaces.
**Fix:** Batch in chunks of 10K: `WHERE sends_this_month > 0 LIMIT 10000`.

### 20. Email address normalization inconsistent
Some places: `.toLowerCase().trim()`, some: `.toLowerCase()`, some: nothing.
**Fix:** Normalize everywhere: `email.toLowerCase().trim()`.

## PERFORMANCE (3)

### 21. No graceful shutdown in worker
**File:** `apps/worker/src/index.ts:71`
`process.exit(0)` on SIGTERM without draining BullMQ workers or Redis. In-flight jobs lost.
**Fix:** `await Promise.all([workers.map(w => w.close()), connection.disconnect()])` before exit.

### 22. Inefficient lead copy loop
See #18 above.

### 23. Unbounded batch update on monthly reset
See #19 above.

## SECURITY (3)

### 24. `CRON_SECRET` uses string equality — timing attack possible
**File:** `apps/web/src/app/api/cron/send/route.ts:14`
**Fix:** `crypto.timingSafeEqual(Buffer.from(auth || ""), Buffer.from(expected))`

### 25. Unsubscribe token not verified as HMAC
Verify tokens are HMAC-signed with expiry, not just `workspace_id + email`.

### 26. No audit log for sensitive workspace changes
No table tracking who changed API keys, plans, domains.
**Fix:** Add `audit_logs` table with `workspace_id`, `user_id`, `action`, `old_values`, `new_values`.

---

## ARCHITECTURE (4)

### 27. No circuit breaker for external email services
Gmail/Microsoft downtime causes campaign-wide failure with no degradation path.
**Fix:** Implement circuit breaker: 3 failures in 60s → open circuit, reject for 5min.

### 28. Worker process suppresses some uncaughtExceptions
**File:** `apps/worker/src/index.ts:23`
Some exceptions swallowed with `return`. Worker appears healthy but may be deadlocked.
**Fix:** Let all uncaughtExceptions exit. Use PM2 auto-restart + health heartbeat.

### 29. Webhook payload types not discriminated
`fireWebhooks(id, event, payload)` doesn't enforce payload shape matches event type at compile time.
**Fix:** Discriminated union: `type WebhookMessage = { event: "send.opened"; data: SendOpenedData } | ...`

### 30. No DB connection health check
No fast-fail middleware for when Supabase is unavailable.
